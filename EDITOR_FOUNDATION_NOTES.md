# React SVG Editor Foundation Notes

## Purpose

This document explains, in a research-style but practical way, how the current SVG editor foundation was built in `Myine`, why the coordinate system works, and how the code evolved from a static visual proof into a minimal interactive editor core.

The current implementation lives in `src/main.jsx`.

## Current Goal of the Editor

The editor is not yet a full circuit simulator. It is a minimal spatial system with six essential properties:

1. A full-screen drawing surface.
2. A stable coordinate space.
3. A visual reference grid.
4. A data-driven rendering loop where clicks create persistent nodes.
5. Direct manipulation of nodes by dragging in SVG space.
6. A component model where each node owns relative connection points called pins.

Those six properties are the foundation required before adding higher-level behaviors such as selection, wires, snapping, and eventually simulation semantics.

## Why SVG Was Chosen

For a circuit editor, the main requirement is not raw pixel painting but structured geometry:

- components have positions
- pins have exact coordinates
- wires are defined by endpoints or paths
- selection outlines are geometric overlays
- zoom and pan are transformations of a world coordinate system

SVG is a strong fit for this because it is inherently coordinate-based. Every shape is described in terms of geometric values such as `x`, `y`, `cx`, `cy`, `width`, and `height`. That means the rendering medium already speaks the same language as the editor model.

If a node in application state says:

```js
{ id: "node-3", x: 240, y: 180 }
```

then rendering that node as:

```jsx
<circle cx={240} cy={180} />
```

is almost a direct translation from data model to view. This is one of the main reasons SVG is a natural substrate for diagram editors.

As the editor evolved, the model became slightly richer:

```js
{
  id: "node-3",
  x: 240,
  y: 180,
  radius: 12,
  pins: [
    { id: "node-3-pin-top", dx: 0, dy: -20, radius: 4 },
    { id: "node-3-pin-bottom", dx: 0, dy: 20, radius: 4 }
  ]
}
```

This is the first step from "drawing dots" toward "representing electronic components."

## Stage 1: Establishing a Full-Screen Canvas

The first milestone was to create a component that occupies the entire viewport:

```jsx
<div style={{ position: "fixed", inset: 0, background: "#111" }}>
  <svg width="100%" height="100%" />
</div>
```

This solves a basic but important problem: the user needs to feel like they are working on a canvas, not inside a small embedded widget.

### Why `position: fixed` and `inset: 0`

The earliest version used `width: 100vw` and `height: 100vh`. That usually works, but `position: fixed` with `inset: 0` is a slightly stronger way to anchor the editor to the viewport itself. It avoids depending on surrounding document flow and makes the editor behave more like an application surface.

This container is the outer shell. It does not define the coordinate system by itself. It only guarantees that the SVG has a predictable amount of screen space available.

## Stage 2: Turning SVG Into a Coordinate Space

The most important technical decision was to use a `viewBox`:

```jsx
viewBox={`0 0 ${viewport.width} ${viewport.height}`}
```

### What `viewBox` Means

The SVG `viewBox` defines an internal coordinate system. It tells the browser:

"Treat this SVG as if it contains a world that starts at `(0, 0)` and extends to `(viewport.width, viewport.height)`."

If the viewport is 1280 by 720, then:

- top-left is `(0, 0)`
- bottom-right is `(1280, 720)`
- the center is approximately `(640, 360)`

Without `viewBox`, an SVG can still render shapes, but the coordinate behavior is more implicit and less suitable as the basis for an editor. With `viewBox`, coordinate intent becomes explicit.

### Why This Matters for an Editor

An editor needs a stable mapping between:

1. model coordinates, stored in React state
2. rendered coordinates, used by SVG elements
3. pointer coordinates, generated by user interaction

Once those three coordinate spaces are made compatible, editor behaviors become composable. The current implementation achieves that by making the SVG itself the authoritative spatial reference.

## Stage 3: Adding the Grid as a Spatial Cue

The grid was implemented using an SVG `<pattern>`:

```jsx
<pattern id="grid" width="20" height="20" patternUnits="userSpaceOnUse">
  <path d="M 20 0 L 0 0 0 20" fill="none" stroke="#222" strokeWidth="1" />
</pattern>
```

and then applied with:

```jsx
<rect width={viewport.width} height={viewport.height} fill="url(#grid)" />
```

### Why Use `<pattern>` Instead of Drawing Every Line

The grid is conceptually infinite repetition of a 20 by 20 cell. A pattern lets us define that cell once and tile it automatically. This is more elegant and more scalable than manually generating line elements across the whole canvas.

### Why `patternUnits="userSpaceOnUse"`

This is a critical detail.

`userSpaceOnUse` means the pattern dimensions are interpreted in the same coordinate system as the SVG drawing space. So a width of `20` means 20 coordinate units on the canvas, not a percentage of the bounding box of the element being filled.

That makes the grid behave like a real geometric reference system rather than a decorative texture.

### Why the Grid Is Subtle

The grid uses `#222` on a `#111` background. The contrast is intentionally low because the grid is a spatial aid, not the main content. Editors often fail visually when the grid competes with the objects.

## Stage 4: Verifying Positioning with a Test Object

The initial validation step was a hardcoded test shape:

```jsx
<circle cx="100" cy="100" r="12" fill="#4ade80" />
```

This served one purpose: prove that a point placed at a chosen coordinate renders exactly where expected.

This kind of test object is important because it isolates coordinate validity from application complexity. Before selection or dragging or wiring, the editor must first answer a simpler question:

"If I say there is an object at `(100, 100)`, does the canvas agree?"

At this stage, the answer was yes.

## Stage 5: Moving from Static Rendering to State-Driven Rendering

The next mutation was conceptual rather than visual.

Instead of hardcoding the circle directly in JSX, the editor moved to React state:

```jsx
const [nodes, setNodes] = useState([
  { id: "node-1", x: 100, y: 100, radius: 12 }
]);
```

and then rendered:

```jsx
{nodes.map((node) => (
  <circle
    key={node.id}
    cx={node.x}
    cy={node.y}
    r={node.radius}
    fill="#4ade80"
  />
))}
```

### Why This Shift Matters

This is the moment the editor stopped being a static graphic and became a system.

The rendered scene is no longer authored directly in JSX. Instead:

- state describes the scene
- rendering reads the state
- interaction mutates the state
- React re-renders the scene

That loop is the foundation of nearly all interactive editors.

## Stage 6: Reading Click Coordinates Correctly

The initial attempt at click logging used simpler browser event properties such as `offsetX` and `offsetY`. Those can work in basic cases, but they are not the most reliable method for an SVG editor.

The current implementation uses this approach:

```jsx
const svg = event.currentTarget;
const point = svg.createSVGPoint();
point.x = event.clientX;
point.y = event.clientY;

const transformedPoint = point.matrixTransform(svg.getScreenCTM().inverse());
```

### Why Browser Coordinates Are Not Enough

Mouse events begin in screen or client coordinates, which are measured relative to the browser viewport. But editor objects live in SVG user coordinates. These are not always the same space.

Even in the current simple editor, making this conversion explicit is valuable. In future versions with zooming, panning, nested transforms, or nontrivial `viewBox` usage, direct use of screen-space event properties becomes error-prone.

### Breakdown of the Conversion

1. `event.clientX` and `event.clientY`

   These give the pointer position in browser client space.

2. `svg.createSVGPoint()`

   This creates a point object that SVG can transform between coordinate systems.

3. `svg.getScreenCTM()`

   This returns the current transformation matrix from SVG user space to screen space.

4. `.inverse()`

   Inverting that matrix gives the transformation from screen space back into SVG space.

5. `point.matrixTransform(...)`

   Applying the inverse matrix maps the click into the canvas coordinate system.

The result is a proper editor-space coordinate, which is exactly what should be stored in state.

### Why the Coordinates Are Rounded

The implementation rounds the transformed coordinates:

```js
{
  x: Math.round(transformedPoint.x),
  y: Math.round(transformedPoint.y),
}
```

This keeps the HUD readable and gives clean integer positions for early editor development. If subpixel precision becomes important later, rounding can be removed.

## Stage 7: Adding a Readout Overlay for Verification

The on-screen coordinate display was added as a debugging instrument:

```jsx
<CoordinateReadout recentChange={recentChange} nodeCount={nodes.length} />
```

This overlay serves two purposes:

1. It confirms spatial behavior without requiring DevTools.
2. It exposes the editor's internal state in a human-readable way.

This is a lightweight form of observability. Early-stage systems benefit greatly from visible state because it shortens the feedback loop between code changes and behavioral understanding.

### Why the HUD Evolved from "Last Click" to "Recent Change"

The earlier version of the HUD displayed the raw coordinates of the last click. That was useful during initial coordinate-system verification, but it became less expressive as the editor gained more behaviors.

Once the editor supported:

- node creation
- node dragging
- pin selection

the most important thing was no longer just "where did the pointer click?" but rather "what was the most recent meaningful state change?"

The HUD was therefore upgraded from a click log into a state-change summary. It now reports actions such as:

- `Added node-4 (320, 180)`
- `Moved node-2 (510, 244)`
- `Selected node-3:pin-top (400, 180)`

This is a better fit for an editor because it reflects mutations to the model, not just low-level input events.

## Stage 8: Mutating the System Into "Click to Create"

The next major behavioral mutation was changing the click handler from a passive logger into a state mutator:

```jsx
setNodes((previousNodes) => [
  ...previousNodes,
  {
    id: `node-${nextNodeId.current++}`,
    x: coordinates.x,
    y: coordinates.y,
    radius: 12,
  },
]);
```

### Why the Functional State Update Is Used

The setter uses:

```js
setNodes((previousNodes) => ...)
```

instead of:

```js
setNodes([...nodes, newNode])
```

because the functional form is safer when multiple updates may occur over time. It guarantees the next state is derived from the latest committed previous state rather than from a stale closure.

### Why This Is the First Real Editor Behavior

This is where the canvas stopped being only a display and became generative:

- the user clicks
- the editor computes a spatial coordinate
- the editor creates a new domain object
- the object is stored in application state
- the view updates to include it

That loop is the minimal interactive editing pipeline.

## Why Nodes Were "Vanishing" Earlier

You observed that after several clicks, older spots seemed to disappear. The most likely cause was unstable or duplicate React keys.

In React, when rendering a list:

```jsx
nodes.map((node) => <circle key={node.id} ... />)
```

the `key` is how React decides whether an element is:

- the same old element
- a moved element
- or an entirely new element

If two different nodes accidentally share the same key, React may reuse or replace a previously rendered element rather than rendering a distinct new one.

### Why Timestamp IDs Were Risky

A common quick approach is:

```js
id: Date.now().toString()
```

That often works, but it is not guaranteed to be unique if events occur quickly enough within the same clock resolution window. In an editor, even rare collisions are unacceptable because spatial persistence is fundamental.

### Why `useRef` Solves It Cleanly

The current code uses:

```jsx
const nextNodeId = useRef(2);
```

and then:

```js
id: `node-${nextNodeId.current++}`
```

This gives:

- stable uniqueness within the session
- no extra re-renders when the counter changes
- a clear separation between render state and internal bookkeeping

`useRef` is appropriate here because the counter is not visual state. It is mutable data that should persist across renders without itself triggering a render.

## Stage 9: Making the Viewport Reactive

The current code stores viewport size in state:

```jsx
const [viewport, setViewport] = useState(() => ({
  width: window.innerWidth,
  height: window.innerHeight,
}));
```

and synchronizes it on resize:

```jsx
useEffect(() => {
  function handleResize() {
    setViewport({
      width: window.innerWidth,
      height: window.innerHeight,
    });
  }

  window.addEventListener("resize", handleResize);
  return () => window.removeEventListener("resize", handleResize);
}, []);
```

### Why This Mutation Was Useful

The early version captured the window dimensions only once. That is enough for a quick prototype, but it becomes inaccurate if the browser window changes size.

By moving viewport dimensions into state:

- the `viewBox` stays synchronized with the actual screen area
- the grid rect continues to cover the visible canvas
- click-to-coordinate mapping remains coherent after resize

This is a small but important stabilization step.

## Stage 10: Adding Direct Dragging Without Separate Mouse State

The latest mutation introduces dragging, but in a deliberately minimal way. The guiding rule was:

"Detect mouse movement, convert it to SVG coordinates, and apply it directly to the dragged node."

This means the editor does **not** store a continuously updating mouse-position state such as:

```jsx
const [mousePosition, setMousePosition] = useState(...)
```

Instead, mouse movement is treated as an event stream whose only job is to update the active node.

### Why No Separate Mouse State Was Added

For this stage of the editor, storing mouse coordinates separately would add state that is not actually needed for rendering the main scene. The real domain state is:

- which node is being dragged
- where that node currently is

The mouse coordinates are only an intermediate value used to compute the node's next position. Once that position is applied, the editor does not need to preserve the raw mouse position.

This is an important design simplification. It keeps the interaction model focused on domain objects rather than on transient input bookkeeping.

### The Shared Coordinate Conversion Helper

The code now centralizes pointer conversion into a helper:

```jsx
function getSvgCoordinates(svg, event) {
  const point = svg.createSVGPoint();
  point.x = event.clientX;
  point.y = event.clientY;

  const transformedPoint = point.matrixTransform(svg.getScreenCTM().inverse());

  return {
    x: Math.round(transformedPoint.x),
    y: Math.round(transformedPoint.y),
  };
}
```

This helper is important for two reasons:

1. It prevents duplication between click handling and drag handling.
2. It enforces a single canonical path for converting browser pointer events into editor-space coordinates.

That means both creation and dragging obey the same geometric rules.

### How Dragging Works Mechanically

Dragging is implemented with three steps:

1. `onMouseDown` on a node identifies which node should move.
2. `onMouseMove` on the SVG converts the pointer into SVG coordinates.
3. The dragged node is updated in state immediately on each move.

The active node is tracked with:

```jsx
const [draggedNodeId, setDraggedNodeId] = useState(null);
```

When the user presses on a circle:

```jsx
onMouseDown={(event) => onNodeMouseDown(event, node.id)}
```

the handler calls:

```jsx
function handleNodeMouseDown(event, nodeId) {
  event.stopPropagation();
  setDraggedNodeId(nodeId);
}
```

The `stopPropagation()` call matters because the background grid also handles clicks for node creation. Without stopping propagation, clicking on an existing node could accidentally trigger the background behavior as well.

### Why the Movement Happens on the SVG, Not on the Node

Once dragging starts, movement is tracked on the parent SVG:

```jsx
onMouseMove={handleSvgMouseMove}
```

This is better than tracking movement only on the circle itself because the pointer can easily move faster than the small node shape. By listening on the SVG canvas, the editor keeps receiving movement events while the pointer remains within the canvas.

The movement handler is:

```jsx
function handleSvgMouseMove(event) {
  if (!draggedNodeId) {
    return;
  }

  const coordinates = getSvgCoordinates(event.currentTarget, event);

  setNodes((previousNodes) =>
    previousNodes.map((node) =>
      node.id === draggedNodeId
        ? { ...node, x: coordinates.x, y: coordinates.y }
        : node
    )
  );
}
```

This is a direct manipulation loop:

- detect movement
- convert to SVG coordinates
- write the coordinates into the dragged node
- re-render

No extra state is inserted between those steps.

### Why This Is a Good Early Drag Architecture

This design has several advantages:

- it is minimal
- it is easy to reason about
- it reuses the same coordinate mathematics as click creation
- it keeps the domain model centered on nodes

It is also future-compatible with improvements like drag offsets, snapping, or constrained movement.

### Why the HUD Still Matters After Dragging

Even after dragging was introduced, the HUD remained important because it exposes the latest editor mutation in a visible way.

Its purpose is now broader than click verification. It tells us:

- the coordinates of the most recently added node
- the latest position of a moved node
- the world-space location of a selected pin

So the overlay now acts as a small state monitor rather than a simple click logger.

### Why Dragging Stops on `mouseup` and `mouseleave`

The SVG now clears drag state with:

```jsx
onMouseUp={handleStopDragging}
onMouseLeave={handleStopDragging}
```

This ensures the editor does not remain stuck in a drag state after the pointer is released or exits the canvas. The stop handler is simple:

```jsx
function handleStopDragging() {
  setDraggedNodeId(null);
}
```

This is enough for the current minimal editor.

### Preventing "Drag End Creates a Node"

One subtle design issue appears when a canvas supports both:

- clicking the background to create nodes
- dragging existing nodes

If the whole SVG handles creation clicks, then a drag can end with a click event that accidentally creates a new node.

The current code avoids that by moving creation handling onto the background grid rectangle:

```jsx
<rect ... onClick={handleCanvasClick} />
```

while the SVG itself handles movement and drag termination.

This separation of responsibilities is a small but important refinement:

- background handles creation
- node handles drag start
- SVG handles drag movement and drag stop

That event architecture reduces interaction conflicts.

## Stage 11: Upgrading Nodes into Components with Relative Pins

The latest conceptual shift is that a node is no longer just a single visual circle. It is now a minimal component-like object with connection points:

```jsx
function createNode(id, x, y) {
  return {
    id,
    x,
    y,
    radius: 12,
    pins: [
      { id: `${id}-pin-top`, dx: 0, dy: -20, radius: 4 },
      { id: `${id}-pin-bottom`, dx: 0, dy: 20, radius: 4 },
    ],
  };
}
```

This is the first explicitly electronics-oriented modeling choice in the editor.

### Why Pins Are Relative, Not Absolute

The most important detail is that the pins are stored with `dx` and `dy`, not with independent global positions.

For example:

```js
{ id: "node-3-pin-top", dx: 0, dy: -20 }
```

does **not** mean "this pin lives at `(0, -20)` on the canvas."

It means:

"This pin is offset from the node center by 0 units in x and -20 units in y."

That design is crucial because it makes the node the spatial owner of its pins.

### Why Relative Pins Are a Better Model

If pins were stored with absolute coordinates, then moving a node would require updating:

- the node position
- the top pin position
- the bottom pin position
- and eventually every other attached sub-element

That is error-prone and makes the data model harder to maintain.

With relative pins, moving the node only requires updating the node's own `x` and `y`. The pins follow automatically during rendering:

```jsx
cx={node.x + pin.dx}
cy={node.y + pin.dy}
```

This gives the editor a local coordinate system inside each component.

### Why This Is the Turning Point Toward Electronics

At earlier stages, the editor was still essentially a generic geometric canvas. Once pins exist as structured sub-elements of a node, the model begins to resemble actual electronic components.

This change enables later features such as:

- wire start and end anchors
- pin hit-testing
- component-specific pin layouts
- modular rendering of different part types

The editor is still minimal, but the ontology has changed:

- before: nodes were isolated objects
- now: nodes are component bodies with connection interfaces

### How Rendering Changed

Nodes are now rendered as grouped SVG structures:

```jsx
<g key={node.id}>
  <circle cx={node.x} cy={node.y} r={node.radius} />
  {node.pins.map((pin) => (
    <circle
      key={pin.id}
      cx={node.x + pin.dx}
      cy={node.y + pin.dy}
      r={pin.radius}
    />
  ))}
</g>
```

The `<g>` element is significant because it acknowledges that a node is no longer a single primitive. It is now a composed visual object made of:

- one body circle
- two pin circles

This grouped rendering pattern is likely to remain important as components become more sophisticated.

### Why Pins Currently Ignore Pointer Events

The current implementation sets:

```jsx
pointerEvents="none"
```

on the pin circles.

This is a temporary but useful decision. At the current stage, pins are only visual structure. They are not yet independently interactive.

Disabling pointer events on pins prevents them from interfering with:

- clicking the background to create nodes
- pressing the main node body to start dragging

Once the editor supports wire creation or pin selection, this decision can be revisited.

### Why Creation Now Uses a `createNode` Factory

Before pins existed, a new node could be created inline:

```js
{
  id: `node-${nextNodeId.current++}`,
  x: coordinates.x,
  y: coordinates.y,
  radius: 12,
}
```

Now that each node has a richer internal structure, creation is centralized:

```jsx
createNode(`node-${nextNodeId.current++}`, coordinates.x, coordinates.y)
```

This has several advantages:

- one canonical node shape
- fewer chances of inconsistent object structure
- easier future expansion when nodes gain labels, types, or more pins

It also turns node creation from an ad hoc object literal into a small domain-level constructor.

## Stage 12: Making Pins Interactive and Adding a Live Wire Preview

Once pins existed visually, the next major step was to make them behave like actual connection points rather than decorative markers.

This happened in two phases:

1. pins became selectable
2. the selected pin began rendering a live preview line toward the pointer

That second step is the first true visual behavior of a wiring system.

### Pin Selection as a Wiring Primitive

The editor introduced:

```jsx
const [activePinId, setActivePinId] = useState(null);
```

This state stores which pin is currently active for a potential connection.

The important design choice is that the active pin is represented by a globally unique string:

```js
`${node.id}:${pin.id}`
```

This matters because pin names are only locally unique inside a node. For example, many nodes may contain:

- `pin-top`
- `pin-bottom`

So the full editor-level identity of a pin must include both the node and the pin.

### Why Pins Were Changed from Passive to Interactive

Earlier, the pins intentionally ignored pointer events because they were only structural visuals. Once the wiring system began, that changed.

Pins now handle clicks directly:

```jsx
onClick={(event) => onPinClick(event, node.id, pin.id)}
```

and the active pin is rendered differently:

- inactive pin: yellow
- active pin: red with a light stroke

This provides the first explicit feedback that a connection origin has been chosen.

### Why the Preview Line Matters

The live line preview is more important than it looks. It establishes a key editor pattern:

"interaction state can produce temporary geometry before a final object exists."

In other words, the user is not yet creating a persistent wire. They are creating an intention to create a wire, and the editor visualizes that intention in real time.

This is the beginning of a distinction between:

- committed objects, such as nodes
- transient interaction visuals, such as wire previews

That distinction is central to professional diagramming and CAD-style tools.

### Reusing Existing Pointer Tracking Instead of Adding Redundant Mouse State

The suggested next step might appear to require a new state such as:

```jsx
const [mousePos, setMousePos] = useState({ x: 0, y: 0 });
```

But the current editor already maintained a live pointer position in SVG coordinates through `pointerPosition`.

So rather than introducing duplicated state, the implementation reused that existing source of truth.

This is a better design because:

- it avoids parallel state that represents the same concept
- it keeps the movement pipeline simpler
- it lets dragging and wire preview share the same coordinate feed

The editor still converts pointer movement into SVG-space coordinates on every `mousemove`, but now that information drives two possible behaviors:

1. moving a dragged node
2. extending the live wire preview from the active pin

### How the Active Pin Position Is Computed

To draw a preview line, the editor must know where the selected pin lives in world space.

That is handled by:

```jsx
function getActivePinPosition() {
  if (!activePinId) {
    return null;
  }

  const [nodeId, pinId] = activePinId.split(":");
  const node = nodes.find((currentNode) => currentNode.id === nodeId);
  const pin = node?.pins.find((currentPin) => currentPin.id === pinId);

  if (!node || !pin) {
    return null;
  }

  return getPinGlobalPosition(node, pin);
}
```

This function does three things:

1. decode the active pin identity
2. locate the node and the local pin definition
3. convert the pin from local offsets into global SVG coordinates

This is an important pattern: wiring relies on resolving local component geometry into world-space endpoints.

### The Shared Pin Position Helper

The code also introduced:

```jsx
function getPinGlobalPosition(node, pin) {
  return {
    x: node.x + pin.dx,
    y: node.y + pin.dy,
  };
}
```

This small helper matters because it formalizes a common rule:

"A pin's world position is the parent node position plus the pin's local offset."

That rule is now reused in:

- pin rendering
- recent-change reporting for pin selection
- wire-preview origin calculation

Centralizing that logic reduces inconsistency and makes the code easier to extend.

### How the Preview Line Is Rendered

Once the active pin position and pointer position are both known, the editor renders:

```jsx
{activePinId && activePinPosition && pointerPosition && (
  <line
    x1={activePinPosition.x}
    y1={activePinPosition.y}
    x2={pointerPosition.x}
    y2={pointerPosition.y}
    stroke="#22d3ee"
    strokeWidth="2"
  />
)}
```

This line is deliberately:

- conditional
- transient
- non-interactive

It only exists while a pin is active, and it disappears as soon as that interaction state is cleared or replaced.

### Why the Line Is Rendered After the Nodes

The preview line is currently rendered after the nodes in the SVG tree. That means it appears visually on top of earlier elements.

For the current prototype, this is acceptable because the line is short-lived and meant to be prominent. In later iterations, the exact z-order can be revisited depending on whether wires should appear above or below components.

### Why No Library Is Needed Yet

At this stage, the interaction model is still simple enough that plain React state is the right tool.

The editor currently needs only a few orthogonal pieces of interaction state:

- `draggedNodeId`
- `activePinId`
- `pointerPosition`

That is not yet complex enough to justify a dedicated interaction-mode or graph-editing library.

A library might become useful later when the editor needs:

- multiple interaction modes
- persistent wires
- undo and redo
- routing logic
- selection sets
- larger graph synchronization

But for the current wire-preview stage, native React state remains the cleaner solution.

## Stage 13: Committing Persistent Wires and Introducing Deletion

The next gap in the editor was that wires only existed as temporary previews. A preview line is useful for interaction feedback, but it is not yet part of the circuit model.

To move beyond that, the editor introduced persistent wire state:

```jsx
const [wires, setWires] = useState([]);
```

This marks an important transition. The editor now contains not only nodes, but also edges between nodes. In graph terms:

- nodes are vertices
- wires are edges

That is the first true circuit-topology structure in the application.

### How a Preview Becomes a Real Wire

The wiring interaction now works as a two-step commit process:

1. click a pin to make it active
2. click a different pin to create a wire between the two

The resulting wire is stored as:

```js
{
  id: "wire-1",
  from: "node-1:pin-top",
  to: "node-2:pin-bottom"
}
```

This is a strong modeling choice because wires do **not** store raw endpoint coordinates. They store pin identities.

### Why Wires Store Pin IDs Instead of Coordinates

This is the same design philosophy used earlier for relative pins.

If a wire stored only two fixed points such as:

```js
{ x1: 100, y1: 80, x2: 200, y2: 220 }
```

then moving a node would require manually updating every attached wire endpoint.

By storing:

- the source pin identity
- the destination pin identity

the editor can always derive wire endpoints from the current node positions.

That means wires automatically stay connected when nodes move.

This is a major invariant:

"Geometry is derived from topology plus current node state."

That is exactly the kind of invariant needed for a scalable circuit editor.

### The Role of `WireLayer`

The code now includes a dedicated rendering layer for wires:

```jsx
function WireLayer({ nodes, wires }) {
  return wires.map((wire) => {
    const fromPin = resolvePinPosition(nodes, wire.from);
    const toPin = resolvePinPosition(nodes, wire.to);

    if (!fromPin || !toPin) {
      return null;
    }

    return (
      <line
        key={wire.id}
        x1={fromPin.x}
        y1={fromPin.y}
        x2={toPin.x}
        y2={toPin.y}
      />
    );
  });
}
```

This layer is important because it keeps wire rendering separate from node rendering. That separation matters more as the editor grows:

- nodes are components
- wires are relationships between components

The rendering model now reflects that conceptual distinction.

### Why `resolvePinPosition` Became More Important

Earlier, `resolvePinPosition` was useful for wire previews and pin selection. With persistent wires, it becomes foundational.

Every committed wire depends on it to determine where its endpoints should be drawn. This means the function now supports:

- active pin preview
- pin selection feedback
- persistent wire rendering

That reuse is a sign that the editor's geometry logic is beginning to stabilize around a few reliable primitives.

### Preventing Duplicate Wires

The implementation also checks whether a wire already exists between two pins before adding a new one.

This matters because duplicate edges between the same endpoints are usually accidental at this stage of an editor. By rejecting them, the interaction remains cleaner and easier to understand.

### Why Clicking Empty Canvas Cancels Wire Preview

Once the editor had both:

- click-to-create-node
- click-to-start-wire

there was a new interaction ambiguity.

If a pin is active and the user clicks empty canvas, what should happen?

The current behavior is:

- cancel the active wire preview
- do **not** create a new node from that same click

This is a cautious interaction choice. It prevents accidental node creation while the user is in the middle of a wiring gesture.

### Adding Node Selection as a Deletion Prerequisite

Deletion requires the editor to know which object is currently the target of removal. For that reason, the editor introduced explicit node selection state:

```jsx
const [selectedNodeId, setSelectedNodeId] = useState("node-1");
```

This selection is updated when a node is pressed. The selected node is also visually highlighted with a stroke.

Selection now serves two purposes:

1. visible editor focus
2. target resolution for deletion

This is the beginning of a broader selection model that can later support:

- property editing
- multi-select
- copy/paste
- keyboard transforms

### How Deletion Works

The editor now listens for `Delete` and `Backspace` at the window level.

When a node is selected and the user presses one of those keys:

1. the node is removed from `nodes`
2. any wires connected to that node are removed from `wires`
3. the active pin is cleared if it belonged to the deleted node
4. drag state is cleared
5. selection is cleared
6. the HUD reports the deletion as the recent change

This cascading cleanup is important because deletion in graph editors is never just about removing a single visual element. Related topology must also be cleaned up.

### Why Attached Wires Are Deleted Automatically

When a node disappears, its pins disappear too. Any wire referencing those pins becomes invalid.

The implementation therefore removes all wires where either endpoint starts with:

```js
`${selectedNodeId}:`
```

This preserves graph consistency. It prevents the editor from storing edges whose endpoints no longer exist.

### Why Deletion Was Added Now

At this point in the editor's evolution, object creation already existed. Without deletion, the workspace can only accumulate state.

Adding deletion is important because it closes the first basic lifecycle loop for nodes:

- create
- move
- connect
- delete

That loop is one of the first signs that the editor is becoming operational rather than purely demonstrative.

## Stage 14: Introducing Interaction Modes to Prevent Accidental Node Creation

Once node creation, dragging, selection, and wiring all coexisted, a serious usability problem appeared:

"Any click on empty canvas could create a node."

That is too aggressive for a real editor. It means actions like:

- deselecting
- canceling wire intent
- finishing a thought with a blank click

could unintentionally add components to the scene.

This is one of the classic UX problems in interactive editors, and the clean solution is to introduce explicit interaction modes.

### The Mode Model

The editor now stores:

```jsx
const [mode, setMode] = useState("select");
```

with three current values:

- `select`
- `create-node`
- `wire`

This is a small change in code, but a large change in interaction design.

### Why Modes Matter

Modes let the same pointer input mean different things depending on user intent.

Without modes:

- the canvas has to guess what a click means

With modes:

- the editor already knows what kind of action the user is trying to perform

That reduces accidental behavior and makes the system easier to extend.

This is the same core idea used in many professional tools:

- design tools
- CAD systems
- 3D editors
- schematic capture software

### What Each Mode Does

The current mode semantics are:

#### `select`

- clicking empty canvas clears selection
- clicking a node selects it and allows dragging
- clicking a pin focuses its location but does not begin wiring

#### `create-node`

- clicking empty canvas creates a node
- clicking existing nodes does not begin dragging
- the mode is intentionally creation-oriented

#### `wire`

- clicking a pin starts or completes a wire
- clicking empty canvas cancels the active wire preview
- node creation is disabled

This division dramatically reduces accidental cross-behavior between editing actions.

### Why Canvas Click No Longer Always Means "Create"

The canvas background handler now begins by checking mode before deciding what to do.

Conceptually, the flow is:

1. if the editor is in `wire` mode and a pin is active, cancel the preview
2. else if the editor is not in `create-node` mode, treat the click as deselection
3. only in `create-node` mode should a background click create a node

This rule is the heart of the UX fix.

### Why Dragging Is Restricted to `select` Mode

The node press handler now ignores drag initiation unless the editor is in `select` mode.

This is important because otherwise the user could still trigger movement while in:

- `create-node`
- `wire`

Restricting drag behavior to `select` mode keeps the editor mentally coherent:

- select mode manipulates existing objects
- create mode adds new objects
- wire mode creates connections

### Why Pins Behave Differently Outside `wire` Mode

Pins remain visible in all modes, but their meaning changes slightly with mode.

In `wire` mode:

- clicking a pin participates in the wiring workflow

Outside `wire` mode:

- clicking a pin only focuses that pin and updates the HUD
- it does not start a wire

This preserves pin visibility and spatial clarity without allowing unintended wiring gestures.

### The Temporary Toolbar

To make the mode system usable, the editor now includes a small toolbar with three buttons:

- `Select`
- `Create Node`
- `Wire`

This is intentionally simple. It is not the final UI, but it is enough to make interaction intent explicit while the editor architecture matures.

### Why Changing Mode Clears Conflicting State

Switching modes now clears or normalizes state where appropriate:

- drag state is cleared on mode change
- active pin state is cleared when leaving `wire` mode

This matters because modes are not only labels. They are operational contexts. Carrying old interaction state across incompatible modes can create confusing behavior.

### Why This Is a Scalable Architectural Change

The mode system is more than a UX patch. It is the beginning of an interaction-state architecture.

Future behaviors can now attach themselves to explicit modes such as:

- component placement
- wire drawing
- label editing
- measurement
- box selection

In that sense, modes are the first step toward a proper editor command system.

## Stage 15: Wire Hover, Selection, and Deletion

Once wires became persistent, a new usability issue appeared: connections existed, but they were still hard to manage precisely.

The editor could:

- create wires
- render wires
- delete wires indirectly by deleting nodes

but it could not yet treat wires as first-class editable objects.

That changed with the introduction of:

- wire hover state
- wire selection state
- direct wire deletion

This is an important UX milestone because it makes connections controllable rather than merely visible.

### New Wire Interaction State

The editor now stores:

```jsx
const [hoveredWireId, setHoveredWireId] = useState(null);
const [selectedWireId, setSelectedWireId] = useState(null);
```

These two states do different jobs:

- `hoveredWireId` is transient and purely pointer-driven
- `selectedWireId` is persistent until changed or cleared

This mirrors the same distinction used in many editors between:

- hover feedback
- committed selection

### Why Hover Matters

Hover is not just decoration. It tells the user:

"This is the object the editor currently thinks you are targeting."

Without hover feedback, wires are visually thin and can feel ambiguous to interact with. Adding hover makes the system feel more responsive and predictable.

In the current editor:

- normal wires are cyan
- hovered wires are orange
- selected wires are red

That color progression makes the interaction state legible at a glance.

### How `WireLayer` Evolved

`WireLayer` is no longer only a passive renderer of committed connections. It is now also the interaction surface for wires.

Each rendered wire line now responds to:

- `onMouseEnter`
- `onMouseLeave`
- `onClick`

This upgrades wires from topology-only objects into selectable editor objects.

### Why Selection Is Separate from Hover

Hover and selection should not be conflated.

Hover answers:

"What is under the pointer right now?"

Selection answers:

"What object is currently the target of editor commands?"

This distinction becomes especially important once keyboard operations exist, because the delete command should affect the selected wire, not merely the hovered one.

### How Wire Selection Fits the Existing Model

When a wire is clicked:

- node selection is cleared
- drag state is cleared
- active pin state is cleared
- the clicked wire becomes the selected wire

This keeps selection singular and avoids ambiguous edit targets.

At the current stage, the editor effectively supports one primary selected object at a time:

- a node
- or a wire

That is a clean and manageable rule for an early editor.

### Keyboard Deletion Now Handles Two Object Types

The deletion effect previously only knew how to delete a selected node. It now has a higher-level behavior:

1. if a wire is selected, delete the selected wire
2. otherwise, if a node is selected, delete the selected node and its attached wires

This ordering matters because selection now spans multiple object classes.

The delete command has therefore evolved from:

"remove selected node"

to:

"remove the currently selected editable object"

That is a more mature editor concept.

### Why Wire Deletion Is Important Even If Node Deletion Exists

Deleting a node already removes attached wires, but that is not enough for normal editing workflows.

Users also need to:

- keep both connected components
- remove only the connection
- revise circuit structure incrementally

Direct wire deletion is what makes that possible.

### Why Empty Canvas Clicks Deselect Wires

The existing canvas-clearing behavior now also clears selected wire state. This is important because selection should be reversible without needing a second object click.

That means:

- click a wire to select it
- click empty canvas in non-create modes to deselect it

This small rule makes the editor feel less sticky and easier to recover from.

### Why This Stage Matters

At this point, the tool now supports a full interaction loop across both nodes and wires:

- create components
- connect them
- move them
- select them
- delete them

That is the beginning of a complete editing grammar rather than a set of isolated demos.

## Stage 16: Preventing Invalid Connections

Once persistent wiring exists, the editor needs more than drawing behavior. It also needs connection rules.

Without constraints, users can create graph states that are technically drawable but logically poor, such as:

- connecting a pin to itself
- connecting two pins on the same node
- creating duplicate wires between the same endpoints

Those cases may look harmless early on, but they become serious problems once the editor begins to support richer circuit reasoning.

### Why a Logic Safety Layer Matters

This stage is important because it separates:

- what the editor is capable of drawing
- what the editor should allow as valid structure

That distinction is the beginning of a real semantic layer.

In other words, not every drawable edge should become a legal circuit connection.

### The New Validation Utilities

The editor now introduces two small utilities:

```jsx
function parseGlobalPinId(globalPinId) {
  const [nodeId, pinId] = globalPinId.split(":");
  return { nodeId, pinId };
}
```

and:

```jsx
function isValidConnection(fromPinId, toPinId, wires) {
  const from = parseGlobalPinId(fromPinId);
  const to = parseGlobalPinId(toPinId);

  if (!from || !to) {
    return false;
  }

  if (from.nodeId === to.nodeId && from.pinId === to.pinId) {
    return false;
  }

  if (from.nodeId === to.nodeId) {
    return false;
  }

  return !wires.some(
    (wire) =>
      (wire.from === fromPinId && wire.to === toPinId) ||
      (wire.from === toPinId && wire.to === fromPinId)
  );
}
```

These functions create a reusable connection-safety layer instead of scattering ad hoc checks through the click handler.

### Why `parseGlobalPinId` Matters

The editor stores pins using a global identity string such as:

```js
"node-2:pin-top"
```

That is convenient for storage and lookup, but validation often needs the structure unpacked into:

- node identity
- pin identity

`parseGlobalPinId` makes that translation explicit and reusable.

### The Three Rules Currently Enforced

The validation layer currently blocks three classes of invalid connection:

#### 1. Same pin to same pin

This rejects a self-edge where the exact same pin is used as both endpoints.

#### 2. Same node to same node

This blocks wires between two pins on the same node.

This rule is technically optional in some editors, but it is a good default here because it prevents the graph from becoming noisy or semantically confusing before more advanced component behavior exists.

#### 3. Duplicate wire in either direction

This prevents:

- `A -> B`
- and `B -> A`

from being treated as two distinct wires when they represent the same connection in the current model.

### Why This Validation Happens Before Commit

The editor performs the validation before adding a new wire to persistent state.

This matters because it keeps invalid topology out of the data model entirely. That is much better than:

- allowing bad state
- then trying to clean it up later

Preventive validation is usually simpler and more reliable than repair logic.

### What the User Sees When a Connection Is Blocked

The current UX response is intentionally simple:

- the active pin is cleared
- the HUD reports `Blocked invalid connection`

This is enough for now because the editor already has a recent-change feedback surface. Later, this could evolve into:

- transient red preview feedback
- inline status text
- error badges

But the key point is that invalid attempts are no longer silent and no longer mutate the graph.

### Why This Is a Major Step Toward Simulation Safety

Circuit simulation depends on structural predictability.

If the graph can accumulate:

- duplicate edges
- meaningless self-connections
- noisy same-node loops

then later stages such as net extraction, validation, or simulation become harder to reason about.

By enforcing connection validity now, the editor begins to protect future computational layers from malformed user interactions.

## Stage 17: Snap to Grid as a Spatial Consistency Rule

Once nodes could be created, moved, connected, and deleted, another subtle but important problem became visible:

- nodes could sit at arbitrary coordinates
- small misalignments accumulated
- wires began to feel visually loose

This is not just an aesthetic issue. In editors like this, spatial consistency affects readability, editing precision, and future routing behavior.

The solution is to add an explicit grid-snapping rule.

### The Snap Utility

The editor now includes:

```jsx
function snap(value, gridSize = 20) {
  return Math.round(value / gridSize) * gridSize;
}
```

This function enforces the rule:

"Editable node positions should resolve to the nearest grid intersection."

That means the grid is no longer only a visual reference. It has become an active constraint on node placement.

### Why Snapping Is Applied to Nodes, Not Pins

The editor applies snapping when:

- creating a node
- dragging a node

but **not** when rendering or updating pins independently.

This is a crucial distinction.

Pins are defined relative to their parent node:

```js
{ id: "pin-top", dx: 0, dy: -20 }
```

They should remain component-local geometry, not separately snapped objects.

If pins were snapped independently, the editor would break a core invariant:

"A pin's position is derived from node position plus local offset."

Preserving that invariant is more important than forcing every sub-element through the snap function.

### Where Snapping Happens

Snapping currently occurs in two places:

#### 1. Node creation

When the user places a node, the raw SVG pointer coordinates are first converted into:

```js
{
  x: snap(coordinates.x),
  y: snap(coordinates.y)
}
```

Only then is the node created.

#### 2. Node dragging

During movement, the drag logic writes snapped coordinates back into node state:

```jsx
x: snap(coordinates.x),
y: snap(coordinates.y)
```

This means node movement is discretized to the grid in real time.

### Why This Is More Than Visual Polish

Grid snapping enforces a shared spatial rhythm across the editor.

That matters because later systems will implicitly depend on predictable placement, including:

- wire routing
- component alignment
- net readability
- layout debugging
- possible simulation visualization

In other words, snapping is the start of a precision system.

### Why Continuous Snapping Is Acceptable for Now

A more premium interaction model would allow:

- smooth free dragging while the pointer moves
- snapping only on release

That can feel more fluid in mature editors.

However, the current implementation chooses continuous snapping during drag because it is:

- simpler
- deterministic
- easy to verify visually

For an early editor foundation, this is a good tradeoff.

### How Snapping Changes the Meaning of the Grid

Before this stage, the grid was primarily:

- a visual cue

After this stage, the grid becomes:

- a placement rule
- a movement constraint
- a structural alignment system

That is a significant conceptual upgrade.

## Stage 18: Expanding the Canvas Beyond the Viewport

The original editor used a canvas sized directly to the visible browser window. That was enough for proving the coordinate system, but it imposed an artificial workspace limit:

- the visible screen area

As soon as the editor began supporting multiple nodes, wires, deletion, snapping, and selection, that limit became too restrictive. A schematic editor needs room to spread out.

### The New Canvas Model

The editor now uses a fixed larger workspace:

```jsx
const canvasSize = { width: 3200, height: 2000 };
```

The SVG is rendered at that full size:

```jsx
<svg
  width={canvasSize.width}
  height={canvasSize.height}
  viewBox={`0 0 ${canvasSize.width} ${canvasSize.height}`}
/>
```

and the outer application shell allows scrolling:

```jsx
overflow: "auto"
```

This means the app still occupies the full browser window, but the drawable world is now larger than the visible area.

### Why This Matters

This is not merely "more space." It changes the editor from:

- a screen-sized demo surface

into:

- a larger navigable workspace

That is much closer to how actual editors behave. Real design and schematic tools almost always distinguish between:

- the viewport
- the canvas or world

Even though this project does not yet support pan or zoom, expanding the canvas is an early approximation of that distinction.

### Why Scrolling Was Chosen

At this stage, the editor does not yet have a dedicated camera or pan system. The simplest way to provide a larger workspace is therefore to let the browser handle movement through scrolling.

This is a practical intermediate step because it:

- increases working area immediately
- does not require additional transformation logic
- keeps the SVG coordinate space simple

Later, this can evolve into a more sophisticated navigation model with panning and zoom.

### Why the Coordinate System Still Works

Even though the canvas is now larger than the viewport, the coordinate model remains consistent because:

- the SVG `viewBox` matches the full canvas dimensions
- node and wire coordinates are still stored directly in canvas space
- pointer events are still converted through the SVG coordinate system

So the editor's geometry rules remain the same; only the available workspace has expanded.

### Why This Is a Meaningful Step

As the editor gains more features, spatial crowding becomes a real usability problem. Expanding the canvas early helps avoid artificial compression and gives future systems more room for:

- larger test circuits
- more wires
- more deliberate layout
- future component palettes or side panels

It is a quiet change, but it materially improves the editor's ability to grow.

## Stage 19: Hitbox System for Pins and Wires

As the editor became more interactive, another practical UX problem emerged:

- pins are visually small
- wires are visually thin

That is visually correct, but not interaction-friendly.

If users have to click with pixel-level precision, the editor feels frustrating even when the underlying logic is correct.

The solution is to separate:

- visual geometry
- interaction geometry

This is the purpose of the hitbox system.

### The Core Principle

The editor now follows an important interaction rule:

"An object's clickable area does not need to match its visible area."

This is standard practice in high-quality interactive tools because precision of appearance and precision of input are not the same design problem.

### Pin Hitboxes

Each pin now has two layers:

1. a larger invisible circle used for interaction
2. a smaller visible circle used for rendering

Conceptually:

```jsx
<g>
  <circle r="12" fill="transparent" onClick={...} />
  <circle r={pin.radius} fill="yellow" pointerEvents="none" />
</g>
```

This is a major usability improvement because the pin remains visually precise while becoming much easier to target.

### Why the Invisible Shape Comes First

The invisible interaction circle is rendered before the visible pin.

That ordering matters because the first circle is intended to own pointer interaction, while the visible circle is intentionally passive:

```jsx
pointerEvents="none"
```

This keeps the interaction model clean:

- the large invisible shape handles clicks
- the small visible shape only communicates state

### Why `fill="transparent"` Is Correct

The invisible hit area uses:

```jsx
fill="transparent"
```

This is important because the shape must remain present in the SVG hit-testing model while not becoming visually opaque.

Using `pointerEvents="none"` on that hitbox would defeat the purpose entirely, because it would stop receiving input.

### Wire Hitboxes

The same design idea now applies to wires.

Each wire is rendered as two layered lines:

1. a wide invisible line used for hover and click interaction
2. a thinner visible line used for actual display

Conceptually:

```jsx
<g>
  <line stroke="transparent" strokeWidth="14" onClick={...} />
  <line stroke="cyan" strokeWidth="2" pointerEvents="none" />
</g>
```

This makes wire targeting dramatically easier without making the diagram look heavy or messy.

### Why This Matters for the Editor's Feel

Hitboxes do not change the circuit model at all. They change something equally important:

- interaction confidence

Once hitboxes exist, users can act with intent rather than hesitation. That matters especially for:

- pin selection
- wire creation
- wire editing
- future hover feedback and tooltips

This is one of those changes that often feels small in code but large in perceived quality.

### Why Hitboxes Are a Structural UX Improvement

The hitbox system is not merely polish. It formalizes a durable editor pattern:

- visual representation can stay precise
- interaction affordance can be forgiving

That principle will remain useful later for:

- labels
- component bodies
- resize handles
- bend points
- tiny tool icons

In that sense, hitboxes are a reusable interaction architecture, not a one-off tweak.

## Stage 20: Fixing HUD and Toolbar Anchoring During Canvas Scroll

Once the canvas expanded beyond the viewport and became scrollable, a subtle UI-layering issue appeared:

- the canvas scrolled correctly
- but the HUD and mode toolbar scrolled with it

That behavior made the controls feel like part of the drawing surface rather than part of the application interface.

### Why This Was a Problem

The overlay panels are not canvas objects. They are editor chrome.

That means they should behave differently from:

- nodes
- pins
- wires
- the grid

If they move with the canvas, the user loses a stable reference point for:

- current mode
- recent change feedback
- selection status
- wire and node counts

In practice, that makes navigation feel less controlled.

### The Fix

The solution was simple but conceptually important:

- keep the canvas inside the scrollable application shell
- anchor the HUD and toolbar to the viewport instead of the canvas

In CSS terms, both overlays were changed from:

```css
position: absolute;
```

to:

```css
position: fixed;
```

### Why `position: fixed` Is the Right Choice Here

`position: fixed` makes an element attach to the browser viewport rather than to the scrolling content area.

That means:

- the canvas can move underneath
- the controls remain visually stable

This is exactly the desired behavior for persistent interface controls in an editor.

### Why This Matters Architecturally

This change reinforces an important separation:

- canvas world
- application UI

Those are different layers of the system and should not be treated as if they belong to the same coordinate space.

The canvas is scrollable, spatial, and content-driven.

The HUD and toolbar are viewport-anchored, informational, and operational.

That distinction becomes more important as the tool grows and gains more interface surfaces.

## Stage 21: Separating Click Intent from Drag Intent

As the editor became more functional, a new interaction-quality issue emerged:

- even a tiny mouse movement after pressing a node could start a drag

That creates a frustrating effect where normal selection clicks can be misinterpreted as motion.

This is a common problem in graphical editors. The solution is to introduce a drag threshold so that movement must exceed a small distance before the editor commits to a drag.

### The New Drag Intent State

The editor now stores:

```jsx
const [dragStart, setDragStart] = useState(null);
const [isDragging, setIsDragging] = useState(false);
```

These states separate:

- the initial pointer-down intent
- the later decision that a drag is truly happening

That is an important distinction. A press is not automatically a drag.

### What Happens on Mouse Down

When the user presses on a node in `select` mode, the editor now records:

- the initial pointer position in screen coordinates
- the node that might become the drag target

But it does **not** begin dragging immediately.

This is the key behavior change.

### The Threshold Rule

During `mousemove`, the editor computes the distance traveled from the original press:

```jsx
const dx = event.clientX - dragStart.x;
const dy = event.clientY - dragStart.y;
const distance = Math.sqrt(dx * dx + dy * dy);
```

Only when that distance exceeds the threshold, currently `5` pixels, does the editor promote the interaction into a real drag.

This is the moment where the system separates:

- click intent
- drag intent

### Why 5 Pixels Is a Good Default

A threshold in the range of 4 to 6 pixels is a common sweet spot because it:

- tolerates natural hand jitter
- still feels responsive
- avoids requiring exaggerated movement

The exact value can be tuned later, but 5 pixels is a sensible baseline for a desktop editor.

### Why This Improves UX So Much

Before the threshold:

- slight motion could cause accidental dragging

After the threshold:

- clicks remain clicks
- tiny jitter is ignored
- deliberate movement begins a drag

That makes the editor feel much more intentional and controlled.

### How This Interacts with Snapping

The drag threshold changes *when* dragging begins, but it does not change *how* position is updated once dragging is active.

Once the threshold is crossed:

- the node still moves in SVG coordinates
- the position is still snapped to the 20px grid

So the editor now has two layers of movement quality:

1. intent filtering before drag begins
2. spatial snapping during actual drag

That combination is a strong early interaction foundation.

### Why Drag State Is Cleared Explicitly

The editor clears:

- `dragStart`
- `draggedNodeId`
- `isDragging`

on mouse release, pointer leave, and mode changes.

This matters because intent-tracking state should never linger after an interaction ends. If it did, later clicks and moves could inherit stale assumptions.

### Why This Stage Matters

This is one of those changes that can make the editor feel "professional" even though the code change is relatively small.

It improves:

- predictability
- selection accuracy
- confidence while editing

In other words, it upgrades raw functionality into controlled interaction.

## Stage 22: Selection Intelligence Through Relationship Awareness

Once nodes and wires could both be selected, another usability limitation became obvious:

- selecting a node only highlighted that node
- selecting a wire only highlighted that wire

This made the editor feel visually isolated, even though the underlying graph already contained relationships.

The next improvement was to make selection context-aware.

### The Core Idea

Selection should reveal structure, not just identity.

That means:

- selecting a node should show which wires are attached to it
- selecting a wire should show which nodes it connects

This is the beginning of relationship-aware visualization.

### Why This Matters

A circuit editor is not only a collection of separate objects. It is a connected graph.

If the editor already knows the relationships internally, then the UI should surface that knowledge. Doing so makes the system feel more intelligent and helps users understand topology at a glance.

This is also an early conceptual bridge toward simulation, because simulation ultimately depends on connection structure.

### New Relationship Helpers

The editor now includes:

```jsx
function getConnectedNodeIdsForWire(wire) {
  const from = parseGlobalPinId(wire?.from);
  const to = parseGlobalPinId(wire?.to);

  if (!from || !to) {
    return [];
  }

  return [from.nodeId, to.nodeId];
}
```

and:

```jsx
function isWireConnectedToNode(wire, nodeId) {
  const connectedNodeIds = getConnectedNodeIdsForWire(wire);
  return connectedNodeIds.includes(nodeId);
}
```

These helpers convert raw wire endpoint IDs into relationship-level information that rendering can use.

### How Node Selection Now Affects Wires

When a node is selected, every wire connected to that node is rendered with a stronger contextual color.

The current visual rule is:

- selected wire: red
- wire connected to selected node: orange
- hovered wire: yellow
- normal wire: cyan

This means node selection now reveals connection fan-out immediately.

### How Wire Selection Now Affects Nodes

When a wire is selected, the editor resolves the two endpoint nodes and highlights them as connected participants.

The current visual rule is:

- actively selected node: orange
- node connected to selected wire: light blue
- normal node: green

This makes the wire feel like a relationship between two real objects rather than a floating line.

### Why `selectedWire` Is Resolved Once in the Editor

Instead of repeatedly searching for the selected wire during every node render, the editor now computes:

```jsx
const selectedWire = wires.find((wire) => wire.id === selectedWireId) ?? null;
```

and passes that result into `NodeLayer`.

This is a small optimization, but more importantly it keeps responsibility clear:

- `Editor` resolves state context
- `NodeLayer` uses that context to render

That is a cleaner pattern than scattering lookup logic deep inside presentation code.

### Why This Stage Matters

This stage is important because it changes selection from:

- object emphasis

into:

- graph-aware emphasis

That is a deeper kind of interface intelligence. The system is no longer merely highlighting what the user clicked; it is showing what that object means in context.

That is exactly the kind of behavior that makes an editor feel capable rather than mechanical.

## Stage 23: Upgrading Wires from Straight Segments to Path-Based Geometry

Until this point, each wire was effectively a straight segment between two endpoints. That was enough to establish connectivity, but it imposed a rigid representation:

- one wire
- one straight line

The next step was to make wires path-based.

### The New Wire Structure

The wire model has been upgraded from a minimal endpoint pair into a richer object:

```js
{
  id: "wire-1",
  from: { nodeId, pinId },
  to: { nodeId, pinId },
  points: []
}
```

This is a major design change because the wire is no longer defined only by two endpoints. It is now defined by:

- a source endpoint
- zero or more intermediate control points
- a destination endpoint

That allows the wire path to bend.

### Why Endpoints and Control Points Are Stored Differently

This stage introduces an important modeling distinction.

Wire endpoints are stored as semantic references:

- `from: { nodeId, pinId }`
- `to: { nodeId, pinId }`

These are not absolute coordinates. They are identities that must be resolved dynamically against the current node positions.

Control points, by contrast, are stored as actual canvas positions:

```js
{ x: 200, y: 150 }
```

This split is exactly the right architecture:

- endpoints stay attached to moving components
- bend points describe absolute path shape

That means wire geometry remains responsive to node movement while still allowing manual shaping.

### Why Endpoint References Became Structured Objects

Earlier, pin endpoints were often treated as encoded strings such as:

```js
"node-2:pin-top"
```

That is still useful as a global identifier in some interaction paths, but committed wires now store structured endpoint objects instead.

This is better for long-term extensibility because endpoint data is now explicitly shaped as:

- `nodeId`
- `pinId`

rather than hidden inside string parsing at every layer of the system.

### Rendering Wires with `<polyline>`

The rendering model has also changed. Instead of treating a wire as a simple `<line>`, the editor now builds a point list:

```jsx
const pathPoints = [fromPin, ...(wire.points ?? []), toPin];
const pointsString = pathPoints.map((point) => `${point.x},${point.y}`).join(" ");
```

and renders:

```jsx
<polyline
  points={pointsString}
  fill="none"
  stroke="cyan"
/>
```

This is a powerful upgrade because it means:

- a wire with no control points still behaves like a straight connection
- a wire with one or more control points bends naturally

So the editor gains path flexibility without needing a separate wire type.

### Why This Is a Good Transitional Design

At the current stage, wires are not yet fully editable bend-by-bend. But the new model already supports that future.

That is good architectural sequencing:

- first change the data model
- then change the renderer
- later add editing behavior for the new structure

This avoids overbuilding and keeps the current milestone clean.

### Bend Points Are Now User-Driven

The earlier version of the path-based system used a default bend point to prove that polylines worked. That has now been replaced by a real interaction flow.

The current wire-routing behavior is:

1. click a pin to start a wire
2. click empty canvas to insert a bend point
3. click another empty location to add another bend point if needed
4. click a destination pin to commit the final wire

This is a much stronger foundation because bend geometry is now shaped by the user rather than hardcoded by the system.

### Why Blank-Space Clicks Became Bend-Point Inserts

In `wire` mode, once a starting pin is active, clicking empty canvas no longer cancels the preview immediately. Instead, it appends a new point into the draft wire path.

That means the wire path can grow like this:

- start pin
- bend point 1
- bend point 2
- destination pin

This interaction makes the editor feel much closer to real routing behavior.

### The Draft Wire Path

To support this, the editor now keeps temporary bend points while a wire is in progress:

```jsx
const [wireDraftPoints, setWireDraftPoints] = useState([]);
```

These points are not committed wires yet. They are transient routing intent.

That distinction matters:

- `wire.points` belongs to a committed wire
- `wireDraftPoints` belongs to the live interaction before commitment

This is a very healthy modeling split.

### How the Preview Works Now

The live preview path is no longer a straight line from the active pin to the pointer. It is now a polyline built from:

```jsx
[activePinPosition, ...wireDraftPoints, pointerPosition]
```

That means the visible preview always extends from the last bend point the user placed, not just from the original pin.

This is exactly the behavior needed for multi-step routing.

### What Happens on Commit

When the user finally clicks a destination pin, the current `wireDraftPoints` array is copied into the committed wire object:

```js
{
  id,
  from,
  to,
  points: wireDraftPoints
}
```

Then the draft state is cleared so the next wire starts fresh.

### Why This Is a Better Foundation

This change is significant because it turns wire bending from:

- a rendering capability

into:

- a genuine user-controlled editing behavior

That opens the path toward future upgrades such as:

- inserting new bend points into existing wires
- dragging control points
- snapping bends
- smarter route cleanup

But the current system stays disciplined by doing only the minimum needed:

- user-defined bends
- path preview
- commit on destination pin

## Stage 24: Dragging Existing Bend Points

Once users could place bend points while routing a wire, the next natural step was to let them reshape those bends after the wire already existed.

That required a new capability:

- committed bend points must become editable geometry

This is the beginning of post-creation wire editing.

### Why Existing Bend Points Need Their Own Interaction Path

Dragging a bend point is not the same as:

- dragging a node
- selecting a wire
- creating a new wire

It is a separate interaction with its own intent:

"I want to reshape this connection path without changing its endpoints."

That distinction matters because wire shape editing should not interfere with node movement or selection behavior.

### Control Points as Editable Handles

The editor now renders visible control-point handles for the currently selected wire.

Each committed bend point gets:

1. a larger invisible hit area for easy targeting
2. a smaller visible control handle

This follows the same hitbox principle already used for:

- pins
- wires

So the editor is reusing one interaction philosophy consistently across different object types.

### Why Only the Selected Wire Shows Handles

Control-point handles are currently rendered only for the selected wire.

That is a good early design choice because it:

- reduces visual clutter
- makes edit intent clear
- avoids covering the canvas with too many small handles

In other words, the wire must first be selected before its geometry becomes directly editable.

### New Drag State for Control Points

The editor now tracks bend-point dragging separately from node dragging:

```jsx
const [draggedControlPoint, setDraggedControlPoint] = useState(null);
```

This state stores which wire and which bend-point index are currently being moved.

That is important because node drag state and bend drag state have different meanings and should not be overloaded into one variable.

### How Bend Dragging Works

When the user presses a control-point hitbox:

1. node-drag state is cleared
2. active pin state is cleared
3. the wire becomes the selected wire
4. `draggedControlPoint` is set

Then, during `mousemove`, if a control point is active, the editor:

- converts the pointer into SVG coordinates
- snaps those coordinates to the grid
- updates the matching entry in `wire.points`

This is a clean pattern because committed geometry is edited in place without changing the wire's endpoints.

### Why Control-Point Dragging Uses Snapped Coordinates

The current editor applies the same grid discipline to bend points that it applies to nodes.

That means bend points are updated with:

- `x: snap(coordinates.x)`
- `y: snap(coordinates.y)`

This keeps wire paths aligned to the same spatial rhythm as the rest of the editor.

That is especially valuable now that wires are beginning to behave like structured layout elements rather than freehand marks.

### How This Fits the Existing Wire Model

This feature works well because the wire model was already upgraded to:

```js
{
  from,
  to,
  points: [...]
}
```

In other words, bend dragging became possible precisely because the earlier path-based wire model already separated:

- semantic endpoints
- editable intermediate geometry

That is a good example of why the previous architectural step mattered.

### Why This Stage Matters

This stage upgrades wires from:

- routable

to:

- shape-editable

That is a substantial jump in editor maturity. Users are no longer limited to the route they happened to create first. They can refine the wire after placement, which is essential for real schematic editing.

At this point, the wire system supports:

- bend insertion

- selection
- deletion

## Stage 25: Wire Integrity Through Soft Segment Constraints

Once bend points became draggable, a new quality issue appeared immediately:

- wires could become messy
- odd diagonal segments appeared easily
- routing quality depended entirely on user precision

That is not wrong in a purely geometric sense, but it produces visually loose diagrams. The next step was therefore to add a lightweight integrity system that helps wires remain orderly while still allowing manual control.

### The Core Idea

A routed wire is now understood as a sequence of segments:

- from endpoint to first bend point
- bend point to bend point
- last bend point to destination endpoint

This means bend dragging is not just moving a point in isolation. It is reshaping adjacent segments.

Once the system is aware of that, it can guide the user toward cleaner routing.

### Soft Constraint Instead of Hard Locking

The editor now applies a soft orthogonal constraint when dragging bend points.

It compares the dragged point against its anchor segment and chooses the dominant axis:

```jsx
const dx = Math.abs(nextPosition.x - anchorPoint.x);
const dy = Math.abs(nextPosition.y - anchorPoint.y);

if (dx > dy) {
  // horizontal tendency
  newY = anchorPoint.y;
} else {
  // vertical tendency
  newX = anchorPoint.x;
}
```

This is not a hard routing system. The user still controls the point, but the editor gently nudges the bend into a cleaner horizontal-or-vertical relationship.

### Why the Previous Point Is Used as the Anchor

The current implementation constrains each dragged bend point relative to the previous point in the wire path.

That previous point may be:

- the wire's source endpoint
- or the immediately preceding bend point

This gives the editor a simple local reference for deciding whether the drag should remain more horizontal or more vertical.

It is a modest rule, but it already improves visual order substantially.

### Why This Counts as Structural Intelligence

Before this stage, the editor simply accepted whatever geometry the user produced.

After this stage, the editor begins to assist.

That is a meaningful shift:

- the user still acts
- the system now guides

This is one of the hallmarks of mature editing tools. They do not merely record input. They help produce cleaner results.

### Modifier Keys as an Escape Hatch

Soft constraints are helpful by default, but users also need a way to opt out.

The editor now allows freeform dragging when either:

- `Alt`
- or `Shift`

is held during bend dragging.

This is a strong interaction pattern because it gives the best of both worlds:

- assisted routing by default
- manual freedom on demand

The current UI also surfaces this behavior directly when a wire is selected by showing a bottom-of-screen hint:

`Hold Alt or Shift to disable constraints for freeform dragging. Double-click a bend point to remove it.`

That small message helps the constraint system stay discoverable instead of hidden.

### Why This Matters for Future Routing Work

This stage does not yet implement full auto-routing, but it introduces the mindset required for it.

The system now:

- understands segment structure
- applies geometric rules during editing
- distinguishes between assisted behavior and free behavior

Those are exactly the kinds of foundations needed before moving into smarter routing systems later.

### Why This Is a Good Intermediate Step

Jumping directly from freeform wires to full auto-routing would be too large a leap.

Soft constraints are a better intermediate step because they:

- improve quality immediately
- keep the model simple
- preserve user control
- reveal where future routing logic should live

That makes this a high-value architectural upgrade, not just a visual tweak.

By supporting direct removal, the wire-editing loop becomes much more complete:

## Stage 27: Building a Connection Graph

Up to this point, the editor has been primarily a geometric system:

- nodes have positions
- pins have coordinates
- wires have paths

That is enough for interaction and rendering, but not enough for reasoning.

The next architectural step is to derive a connection graph from the committed wires.

### The Core Shift

This stage changes the editor from:

- a drawing system

toward:

- a thinking system

That is because simulation, validation, and net analysis are all fundamentally graph problems.

### Pin Keys as Graph Vertices

The graph is built at the pin level, not at the node level.

Each pin is assigned a stable key:

```jsx
function getPinKey(nodeId, pinId) {
  return `${nodeId}:${pinId}`;
}
```

This means the graph's vertices are electrical terminals rather than whole components.

That is the correct level of abstraction for circuit connectivity.

### Building the Adjacency Graph

The editor now includes:

```jsx
function buildGraph(wires) {
  const graph = {};

  wires.forEach((wire) => {
    const from = parsePinReference(wire.from);
    const to = parsePinReference(wire.to);

    if (!from || !to) {
      return;
    }

    const a = getPinKey(from.nodeId, from.pinId);
    const b = getPinKey(to.nodeId, to.pinId);

    if (!graph[a]) {
      graph[a] = [];
    }

    if (!graph[b]) {
      graph[b] = [];
    }

    graph[a].push(b);
    graph[b].push(a);
  });

  return graph;
}
```

This produces an adjacency-list representation, where each pin key maps to the pins directly connected to it.

That makes graph traversal straightforward.

### Why the Graph Is Derived from Wires Instead of Stored Separately

The graph is not currently stored as independent long-lived state. It is derived from `wires`.

That is a good choice at this stage because:

- wires are already the source of truth for committed connectivity
- the graph can be rebuilt deterministically
- there is no risk of the graph and wire state diverging

This keeps the architecture simple and trustworthy.

### Breadth-First Search as the First Traversal Primitive

The editor now also includes a minimal connectivity query:

```jsx
function isConnected(graph, start, target) {
  const visited = new Set();
  const queue = [start];

  while (queue.length > 0) {
    const current = queue.shift();

    if (current === target) {
      return true;
    }

    if (visited.has(current)) {
      continue;
    }

    visited.add(current);
    const neighbors = graph[current] || [];
    queue.push(...neighbors);
  }

  return false;
}
```

This is a small but very important step because it proves the editor can do more than draw connectivity. It can inspect it.

### Why This Matters for Simulation

Circuit simulation ultimately depends on understanding paths through a network.

The graph foundation now makes it possible to ask questions such as:

- what is connected to this pin?
- is there a path between two pins?
- which pins belong to the same connected region?

Those are the beginnings of electrical reasoning.

### Why the HUD Now Shows Graph Size

The editor now exposes:

- the number of graph pin vertices

in the HUD.

This is a lightweight way of making the logical layer visible while the editor is still in a foundational stage. It confirms that the graph is being derived and updated as the user edits the circuit.

### Why This Stage Is So Important

This is one of the biggest conceptual shifts in the project so far.

Until now, most of the system has been about:

- where things are
- how they look
- how they can be manipulated

Now the editor also starts to know:

- what is connected to what

That is the first true step toward simulation.

## Stage 29: Drag-to-Scroll Canvas Navigation in Select Mode

Once the workspace became larger and zoomable, scrolling by scrollbar alone was no longer enough for comfortable navigation.

The editor now supports an alternate navigation gesture:

- in `select` mode
- click empty canvas
- drag the background
- the canvas scrolls with your movement

This is essentially grab-to-pan behavior built on top of the scrollable workspace.

### Why This Matters

Scrollbars are functional, but they are not always the most natural way to move around a large editing surface.

Dragging empty space is often faster and more spatially intuitive, especially when:

- the user is already working with the mouse inside the canvas
- the circuit extends well beyond the visible viewport
- zoom has changed how much of the workspace is visible

This makes navigation feel more direct.

### Why It Only Works in `select` Mode

The background-drag scroll behavior is intentionally restricted to `select` mode.

That is important because empty-canvas gestures already mean different things in other modes:

- `create-node` mode uses blank clicks for placement
- `wire` mode uses blank clicks for bend insertion

If canvas dragging were active in all modes, it would conflict with those interactions.

Restricting it to `select` mode preserves a clean mode-based interaction grammar.

### How It Works

When the user presses on empty canvas in `select` mode, the editor records:

- pointer start position
- container scroll position

If the pointer then moves beyond a small threshold, the editor begins panning by updating:

- `scrollLeft`
- `scrollTop`

on the scroll container itself.

This means the editor is still using the browser's scroll mechanics, but the user can now drive them through a drag gesture.

### Why There Is a Threshold

Just like node dragging, background panning also uses a small threshold before it is treated as an actual drag.

That prevents simple clicks on empty canvas from being misread as scroll gestures.

It also allows selection clearing to remain a clean click action when no real panning happened.

### Why Canvas Click Handling Needed a Guard

Once drag-to-scroll was introduced, another subtle issue appeared:

- a panning gesture could end with a click event

If not handled carefully, that would cause the canvas click handler to run after a pan and accidentally clear selection or trigger other background behavior.

To prevent that, the editor now suppresses the follow-up canvas click after a real background-drag scroll.

This keeps navigation and click semantics cleanly separated.

### Why This Is a Good Intermediate Navigation Layer

The project still does not yet have a full camera system with:

- true pan transforms
- pointer-centered zooming
- trackpad gesture handling

But drag-to-scroll is a strong intermediate step because it:

- improves navigation immediately
- works with the existing scroll container
- does not require rewriting the coordinate model

That makes it a pragmatic upgrade for the current architecture.

## Stage 33: Deriving Nets from the Connection Graph

The next architectural limitation appeared once the editor began supporting richer circuit structure.

The graph already knew which pins were connected, but the logic was still shaped around pairwise questions such as:

```js
isConnected(A, B)
```

That works for tiny examples, but it becomes awkward as soon as the circuit contains:

- multiple wires
- branching
- shared ground
- multiple LEDs

### The Core Shift

Instead of repeatedly asking whether one terminal can reach another, the editor now derives nets from the graph.

A net is simply a connected component of pin keys.

If these are electrically continuous:

- `battery-1:positive`
- `led-1:anode`
- `node-7:pin-top`

then all of them belong to the same net.

### `buildNets(graph)`

The editor now walks the adjacency graph and groups connected pins together:

```js
function buildNets(graph) {
  const visited = new Set();
  const nets = [];

  for (const pin in graph) {
    if (visited.has(pin)) continue;

    const net = [];
    const stack = [pin];

    while (stack.length) {
      const current = stack.pop();

      if (visited.has(current)) continue;
      visited.add(current);
      net.push(current);

      const neighbors = graph[current] || [];
      stack.push(...neighbors);
    }

    nets.push(net);
  }

  return nets;
}
```

This produces structures like:

```js
[
  ["battery-1:positive", "led-1:anode"],
  ["led-1:cathode", "battery-1:negative"]
]
```

### `findNet(...)` and Net Membership

Once nets exist, logic can ask:

- which net contains this pin?
- do these two pins belong to the same electrical group?

That is now expressed through:

- `findNet(nets, pinKey)`
- `arePinsInSameNet(nets, firstPinKey, secondPinKey)`

### LED Logic Now Runs on Nets

The LED helper now checks whether:

- battery positive and LED anode share a net
- LED cathode and battery negative share another net

That is much closer to circuit reasoning than repeated point-to-point traversal.

It also scales naturally to branches. If battery positive fans out to multiple LEDs, those LEDs can resolve against the same source net without extra special-case logic.

### HUD Feedback Now Includes Net Count

The editor HUD previously exposed:

- node count
- wire count
- graph pin count

It now also exposes the number of derived nets.

That is useful because nets are now a first-class logical structure rather than an implicit side effect of traversal.

### Why This Stage Matters

This stage is important because it upgrades the logical layer from:

- pairwise graph reasoning

to:

- network-level reasoning

That is a major step toward scalable simulation, net labeling, connected-group highlighting, and stronger electrical validation.

## Stage 34: Power Propagation Through Nets

- the editor knew which terminals belonged to the same electrical group

At that point, the logic layer had become richer than the UI layer.

### The Core Idea

If a net contains a battery's positive terminal, that net is now treated as powered.

### `getPoweredNetIndexes(...)`

This new helper turns battery sources into network-level power markers.

That is a meaningful architectural shift because the source of truth is now:

- component semantics
- projected onto
- derived net structure

At the time this stage was introduced, powered terminals were exposed through a small helper over powered nets. In the current implementation, the render layer instead reads powered pins from the derived `simulationState`, while powered nets remain available as a separate topological/debug signal.

### Powered Wires

Committed wires now check whether either endpoint belongs to a powered net.

That matters because it makes propagation feel spatial instead of abstract.

### LED Logic Now Separates "Powered" from "Closed Loop"

The visual layer can still show powered terminals and wires through powered-net highlighting.

## Stage 35: Switch Components as Controlled Internal Connections

The next important step was to introduce a component whose behavior changes connectivity itself.

A switch is not only a drawable symbol. It is a component that conditionally connects two of its own terminals.

### The Switch Model

The switch now follows this shape:

```js
{
  type: "SWITCH",
  state: "OPEN",
  pins: [
    { id: "a", role: "terminal" },
    { id: "b", role: "terminal" }
  ]
}
```

In the current editor, those terminals are laid out horizontally so the switch behaves like a left-to-right controlled bridge.

### The Important Design Decision

The switch does not create hidden internal wires in editor state.

Instead, it mutates the derived connection graph itself.

That is a very strong architectural choice because it keeps the source of truth simple:

- visible wires remain visible wires
- switch conduction is projected directly into graph derivation

### `buildGraph(wires, nodes)`

The graph builder now consumes both:

- committed `wires`
- current `nodes`

It still adds normal wire edges first.

Then it adds an additional internal edge for every switch whose `state` is `CLOSED`:

- `switch-id:a`
- `switch-id:b`

When a switch is `OPEN`, that internal edge is omitted.

### What This Means Structurally

An open switch is now graph-inert.

A closed switch behaves like an internal wire segment for the purposes of:

- nets
- powered-net propagation
- LED evaluation

That is exactly the right level to model controlled continuity.

### Toggle Interaction

Switches can now be toggled directly in `select` mode.

Clicking the switch body flips:

- `OPEN -> CLOSED`
- `CLOSED -> OPEN`

The node state changes, the graph is rebuilt, nets are re-derived, and powered rendering updates automatically.

### Why Click Suppression Matters

Switches are also draggable objects, which means toggle clicks and drag gestures can easily conflict.

To avoid accidental toggles after dragging, the editor now suppresses the post-drag click that browsers normally emit after a drag interaction.

That keeps the switch behavior deliberate instead of fragile.

### Visual Identity

The switch now renders with a dedicated internal contact line:

- `OPEN` uses a gray tilted contact
- `CLOSED` uses a yellow straight contact

This gives the component an immediately readable state even before the user follows the wires around it.

### Why This Stage Matters

This stage is important because it upgrades the editor from:

- static circuits whose topology changes only through wire edits

to:

- interactive circuits whose topology can mutate through component state

That is a major turning point. The editor now supports dynamic graph mutation, re-evaluation, and controlled connectivity as part of normal interaction.

## Stage 36: Separating Topology Graphs from Simulation Graphs

Once switches, buttons, nets, and powered-net feedback all existed, the next limitation became structural:

- the editor had a graph for topology
- but it did not yet have a graph for component behavior

That distinction matters because wire continuity is not the same thing as simulated conduction through components.

### The Core Decision

The editor now treats simulation as a separate graph layer rather than overloading the topology graph with every behavioral rule.

The system now has:

- a connection graph for topology and net derivation
- a simulation graph for directed and conditional traversal behavior

### `buildGraph(wires, nodes)`

The existing graph is still used for structural editor logic.

It includes:

- bidirectional wire edges
- bidirectional internal switch edges when closed
- bidirectional internal button edges when pressed

That graph remains the basis for:

- net extraction
- powered-net derivation
- structural relationship logic

### `buildSimulationGraph(nodes, wires)`

The simulation graph extends traversal behavior by injecting component-internal conduction rules.

It includes:

- bidirectional wire edges
- bidirectional switch edges when `CLOSED`
- bidirectional button edges when `CLOSED`
- directed LED edges from `anode -> cathode`

This is the key shift:

- LEDs are no longer validated outside the graph
- LEDs now participate directly in traversal

### Why This Stage Matters

This stage is important because it upgrades the engine from:

- topology-only graph reasoning

to:

- layered topology-plus-behavior graph reasoning

That is the right foundation for a scalable simulator. Nets remain useful for structure, while the simulation graph becomes the place where component conduction rules live.

## Stage 37: Momentary Buttons as Temporal Connectivity

Switches introduced controlled topology.

Buttons take the next step by introducing controlled topology over time.

### The Button Model

The button now follows the same two-terminal pattern as the switch:

```js
{
  type: "BUTTON",
  state: "OPEN",
  pins: [
    { id: "a", role: "terminal" },
    { id: "b", role: "terminal" }
  ]
}
```

The important difference is behavioral:

- switch state persists until toggled again
- button state persists only while pressed

### Graph Integration

Buttons reuse the same graph-level idea as switches.

When a button is `CLOSED`, `buildGraph(wires, nodes)` injects an internal edge between:

- `button-id:a`
- `button-id:b`

When the button is `OPEN`, that internal connection disappears from the derived graph.

### Press and Release Semantics

Buttons now behave like this:

- `mousedown` on the button body closes it
- `mouseup` reopens it

This means button state is temporal rather than toggle-based.

### Why the Global Mouse-Up Listener Matters

If the user presses a button, drags away, and releases outside the canvas, the component must still return to `OPEN`.

To make that reliable, the editor now listens for `mouseup` on `window` and resets all pressed buttons.

That keeps the interaction believable and prevents stuck button states.

### Visual Identity

The button now renders with its own pressed/unpressed treatment:

- `OPEN` appears gray
- `CLOSED` appears yellow and energized

This makes momentary interaction visible even before the rest of the circuit reacts.

### Why This Stage Matters

This stage is important because it upgrades the editor from:

- persistent topology changes only

to:

- temporal topology changes driven by user input duration

That is a major capability jump. The circuit is no longer only editable. It is now interactable over time.

## Stage 38: Multi-Path Activation Through Reachability Intersection

Once the simulation graph existed, the next limitation became clear:

- a single valid path is enough for simple series circuits
- but it is not enough for parallel circuits where multiple valid conductive routes exist

The simulation therefore needed to move from single-path validation to active-region analysis.

### The Core Shift

Instead of selecting one route from battery positive to battery negative, the editor now computes:

- everything reachable forward from battery positive
- everything that can reach battery negative

The intersection of those two sets becomes the active simulation region.

### `collectReachableNodes(graph, startPinKey)`

This helper performs graph traversal from a starting pin and collects every reachable terminal in that direction.

It is used as the primitive for reachability rather than for choosing a single preferred path.

### `buildReverseGraph(graph)`

To determine which terminals can reach battery negative, the editor constructs a reversed view of the simulation graph.

That lets the same reachability traversal be reused in the opposite direction.

### `getActiveNodesFromBattery(simulationGraph, batteryNode)`

This helper now computes the active simulation set by intersecting:

- forward reachability from `battery:positive`
- reverse reachability from `battery:negative`

Any pin that appears in both sets lies on at least one valid directed route between source and return.

### LED Logic Now Uses Active Simulation Participation

LED activation is no longer derived from:

- powered nets
- local battery checks
- one chosen route

Instead, an LED is on only if:

- its `anode` is active
- its `cathode` is active
- the simulation graph contains the directed internal transition `anode -> cathode`

That means multiple LEDs in parallel can all turn on naturally, while series circuits continue to work without special cases.

### `hasDirectedEdge(...)` and `isLedActive(...)`

These helpers make the final check explicit:

- the LED must participate in the active simulation region
- the LED must still preserve forward conduction internally

This keeps the simulation purely graph-based and directional without introducing electrical quantities such as voltage or current.

### Why This Stage Matters

This stage is important because it upgrades simulation from:

- one-path validation

to:

- multi-path activation on a behavior-aware graph

That is the point where the simulator begins to support parallel circuits naturally instead of only serial ones.

## Stage 39: Behavior Layer for Component Conduction

As the simulator gained more components, another architectural problem started to emerge:

- simulation behavior was still at risk of spreading into type-specific branches
- every new component threatened to add more one-off logic to graph construction

That direction does not scale.

### The Core Shift

The simulator now leans on a behavior layer instead of asking ad hoc questions like:

- is this node a switch?
- is this node a button?
- is this node an LED?

The more useful question is:

- what internal conductive behavior does this component expose?

### `COMPONENT_BEHAVIORS`

The editor now centralizes simulation behavior in a registry keyed by component type.

Each behavior can define:

- topology-layer internal connections
- simulation-layer internal connections
- activation segments
- source pins

That turns component rules into data-driven graph contributions instead of scattered conditionals.

In the current implementation, nodes also carry a `behavior` reference directly. That keeps the existing React state shape compatible while allowing graph builders and simulation helpers to query behavior from the node itself.

### Why This Matters

The behavior layer means new parts can be added by defining:

- how they connect structurally
- how they conduct during simulation
- how they report activation-relevant segments

For the current battery behavior, the simulation layer also exposes a temporary conceptual `positive -> negative` internal connection. That is a phase-specific logical convenience and can later be refined when voltage and source modeling are introduced.

That is a much stronger foundation for future components such as diodes, resistors, and transistors.

### Why This Stage Matters

This stage is important because it upgrades the simulator from:

- type-based branching

to:

- behavior-based graph construction

That is a maintainability milestone as much as a simulation milestone. The engine is becoming extensible instead of merely growing.

## Stage 40: Derived Simulation State as a Render Contract

Once the behavior layer and multi-path activation both existed, the next architectural step was to stop letting render logic infer simulation details independently.

The UI needed one derived object that says:

- which pins are currently powered
- which nodes are currently active

without mutating the nodes themselves.

### The Core Shift

The editor now computes a dedicated derived simulation layer:

```js
{
  poweredPins: Set(),
  activeNodes: Map()
}
```

This object is derived from the simulation graph and passed into rendering.

### `computeSimulationState(graph, nodes)`

This helper now performs the whole logical propagation pass.

It computes:

- `forwardReachable` from battery positive
- `backwardReachable` from battery negative
- `poweredPins` as the intersection of those two sets
- `activeNodes` as per-component derived state

This means the simulation result is calculated once and then consumed everywhere else.

### Powered Pins

A pin is now considered powered only if:

- it is reachable from battery positive
- and it can still reach battery negative

That is a stronger notion than raw topological membership because it comes from the simulation graph rather than the net graph.

### Active Nodes

Node activity is now stored separately from node authoring state.

For LEDs:

- `ON` if `anode` is in forward reachability and `cathode` is in backward reachability

For switches and buttons:

- their open/closed visual meaning still comes from their own state

For other components:

- activity is derived without mutating the original node objects

### Why This Matters

The important benefit is separation:

- editor state stays declarative and user-authored
- simulation state stays derived and disposable

That makes the system easier to reason about, easier to debug, and much safer for future upgrades.

### Why This Stage Matters

This stage is important because it upgrades rendering from:

- scattered simulation lookups

to:

- a single derived simulation contract

That is the point where the UI stops re-solving physics locally and starts reading from one coherent simulation result.

## Stage 46: Introducing a Dirty Flag System for Performance Optimization

As the editor grew in complexity and the simulation logic became more intricate, recomputing the entire circuit graph and simulation state on every single UI update (e.g., mouse movements, minor visual changes) led to performance bottlenecks. To address this, a "dirty flag" optimization system was introduced to cache expensive computations and only re-run them when the underlying circuit topology or component states genuinely changed.

### The `createSimulationEngine` Module

A new, framework-agnostic JavaScript module, `simulationEngine.js`, was created to encapsulate the dirty flag logic and caching mechanism. It exports a factory function `createSimulationEngine` that takes `buildGraph` and `computeSimulationState` as dependencies.

The core of this module manages:
- `isGraphDirty`: A boolean flag indicating whether the circuit's topological graph needs to be rebuilt.
- `cachedGraph`: Stores the last computed graph.
- `cachedSimulationState`: Stores the last computed simulation results.

It exposes three methods:
- `markDirty()`: Sets `isGraphDirty = true`, invalidating the cached graph and forcing a re-computation on the next request for data.
- `getGraph(nodes, wires)`: Returns the cached graph if `isGraphDirty` is `false`. If `isGraphDirty` is `true`, it first rebuilds the graph, updates `cachedGraph`, sets `isGraphDirty = false`, and then returns the new graph.
- `getSimulationState(nodes, wires)`: Internally calls `getGraph` to ensure the graph is up-to-date, then uses the (potentially new) graph to `computeSimulationState`. The result is cached in `cachedSimulationState` before being returned.

### Integration with React (`main.jsx`)

1.  **Engine Initialization**: The `createSimulationEngine` is instantiated once using `useMemo` with an empty dependency array (`[]`) in the `Editor` component. This ensures the engine instance (and its internal cache) persists across renders:
    ```jsx
    const engine = useMemo(
      () => createSimulationEngine(buildGraph, computeSimulationState),
      []
    );
    ```

2.  **Memoized Computations**: The `connectionGraph` and `simulationState` are now derived using `useMemo` and rely on the `engine`'s `getGraph` and `getSimulationState` methods, respectively. Their dependency arrays include `nodes`, `wires`, and `engine` to ensure React re-evaluates these memos only when these props change, or when the `engine` instance itself might hypothetically change (though it's stable via its own `useMemo`).
    ```jsx
    const connectionGraph = useMemo(() => engine.getGraph(nodes, wires), [nodes, wires, engine]);
    // ...
    const simulationState = useMemo(
      () => engine.getSimulationState(nodes, wires),
      [nodes, wires, engine]
    );
    ```

3.  **`markDirty()` Triggers**: The `engine.markDirty()` method is strategically called within event handlers that modify the circuit's *logical* or *topological* structure:
    *   **Wire added or removed**: In `handleKeyDown` (for deleting wires) and `handlePinClick` (for creating new wires). Changes to `wires` directly affect connectivity.
    *   **Node added or removed**: In `handleKeyDown` (for deleting nodes) and `handleCanvasClick` (for creating new nodes). Changes to `nodes` fundamentally alter the circuit's components and their connections.
    *   **Node state changes (e.g., switch/button toggle)**: In `handleNodeClick` (for toggling switches) and `handleNodeMouseDown`/`handleGlobalMouseUp` (for pressing/releasing buttons). These actions change how components conduct electricity, effectively modifying the graph's internal edges.
    *   **Node moved**: **Does NOT** call `markDirty()`. Moving a node only changes its `x` and `y` coordinates for rendering. The circuit's electrical connections (its topology) remain unchanged, so a graph rebuild is unnecessary. This is a significant optimization for drag operations.

### Performance Benefits and Stale-State Prevention

This dirty flag system dramatically improves performance by:
-   **Avoiding unnecessary recomputations**: Expensive graph building and simulation logic is skipped if only visual properties (like node position during a drag) change.
-   **Leveraging React's `useMemo`**: React efficiently re-runs memoized functions only when their dependencies change, further reducing overhead.

A key "stale-state pitfall" (where old cached results are incorrectly used after a relevant state change) is prevented by:
-   **Carefully identifying all topological/conductive changes**: Ensuring `markDirty()` is called for *every* action that alters the circuit's electrical behavior.
-   **Centralized cache invalidation**: `markDirty()` directly invalidates the cache, forcing `getGraph` and `getSimulationState` to re-run their expensive operations on the next access.

## Stage 41: Diagnostics Layer for Circuit Validation

With a stable simulation engine in place, the next requirement was to provide user feedback regarding circuit correctness without modifying the core simulation logic.

The diagnostics layer acts as a separate analysis pass that consumes the derived simulation state and topology to produce a structured summary of potential issues.

### The Diagnostic Object

The output follows a strict structure:
```js
{
  inactiveNodes: [],
  partialNodes: [],
  floatingGroups: [],
  hasOpenCircuit: boolean,
  hasShortCircuit: boolean
}
```

## Stage 42: Detecting Inactive and Partially Connected Components

The first diagnostic tasks involve identifying components that are not participating effectively in the active circuit.

### Inactive Components
A node is flagged as inactive if **none** of its pins are present in the `poweredPins` set. This identifies components that are completely disconnected from the current conductive loop.

### Partially Connected Components
This catches cases like an LED with only one leg wired. A node is flagged if it is not `active` (conducting) but at least one of its pins is present in either the `forwardReachable` or `backwardReachable` sets. This indicates that power or ground has reached the component, but the path is not closed.

## Stage 43: Identifying Floating Subgraphs

A "floating circuit" occurs when multiple components are connected to each other but have no electrical path back to a battery terminal.

### The Detection Algorithm
1. **Grouping:** The system uses a disjoint-set approach to group nodes into connected clusters based on shared `nets`.
2. **Validation:** For each cluster, the system checks if **any** pin in the entire group is reachable from the battery (forward or backward).
3. **Flagging:** Groups that have zero reachability to the source or return are returned as `floatingGroups`.

This is a significant logical check because it identifies "TEST" node clusters or isolated logic that won't function due to a lack of power reference.

## Stage 44: Open Circuit Detection

While reachability intersection handles parallel paths, users need to know if a primary loop exists at all.

### The Rule
If a `BATTERY` exists in the circuit, the system checks if its `negative` terminal is present in the `poweredPins` set. 

If the battery is present but its negative terminal is not "powered," it means the forward flow from positive never successfully navigated through the components back to the return. This triggers `hasOpenCircuit: true`.

## Stage 45: Basic Short Circuit Detection

Short circuits are detected by comparing topological continuity against component behavior.

### The Logic
A short circuit is defined as a low-resistance path directly between battery terminals. Since the `nets` (topology graph) specifically exclude paths through resistive/directional components like LEDs, a short circuit is present if:

**The Battery's `positive` and `negative` pins belong to the same electrical net.**

This effectively detects if a wire (or a series of closed switches) creates a direct bridge between the two battery terminals, bypassing the intended load.

## Architectural Pattern That Emerged

Even though the code is still small, it already follows a useful structure:

1. Definitions
   `GridPattern`

2. Pure render layers
   `NodeLayer`

3. Debug or instrumentation UI
   `CoordinateReadout`

4. Stateful orchestration and event coordination
   `Editor`

This is a healthy direction for future growth because it separates:

- static visuals
- dynamic scene rendering
- interaction logic
- developer-facing observability

## Why This Foundation Matters for a Circuit Editor

Circuit editors are fundamentally spatial editors with semantic overlays.

At a minimum, they need:

- coordinate-accurate placement
- stable identity for objects
- scene re-rendering from state
- reliable pointer-to-world conversion
- direct manipulation of spatial objects
- component-local connection geometry
- transient connection previews before a wire is committed

The current implementation already provides those primitives.

That means future features can build on established invariants:

- a node's position in state is meaningful
- a click maps to a real canvas coordinate
- a drag maps to a real canvas coordinate
- rendering is derived from data rather than ad hoc DOM mutation
- each node has a persistent identity
- each pin is positioned relative to its parent node
- an active pin can be resolved into a world-space wire origin
- the pointer can drive temporary geometry in real time

Those invariants are more important than any early visual polish.

## Stage 50: Simulation Debug Overlay

To assist in debugging complex circuits, a dedicated simulation debug overlay was introduced. This layer visualizes the internal conductive paths and blocking conditions that the engine calculates during simulation.

### Traversal Analytics
The `computeSimulationState` function was expanded to return a detailed report of the traversal state, including `poweredNodes` and specific `pathEdges`.

### Blocking Conditions
The engine now explicitly identifies primary blocking conditions:
1. **Open Contacts**: Switches or Buttons in the `OPEN` state.
2. **Reverse Biased Diode**: LEDs where the potential flow direction is from cathode to anode.

### Visual Feedback
The `SimulationOverlay` provides high-visibility feedback: Path Edges are emphasized in yellow, while Blocked Edges are rendered as red dashed lines.

That sequence is small, but it is exactly the right order for building an editor foundation without overbuilding.
